# FL-05 — Agent and MCP Basics

An automation workflow follows a known route. An agent works toward a goal while deciding which route to take. That distinction matters because reliable work usually needs both: a predictable core and limited judgment around it.

The FL-04 portfolio-readiness workflow is automation. Its sequence is fixed: read one artifact, test three gates, return `READY` or `REVISE`, and require human approval. The same prompt structure is reused for every run. This makes the output easy to compare and audit. If I turned it into an agent, I would give it the broader goal “prepare these Week 4 artifacts for submission.” It might inspect the folder, choose which artifact to review first, open evidence links, revise a file, publish it, and then submit the correct URL. Those decisions are useful, but each adds risk. For this assignment, the review loop should remain deterministic while an agent may coordinate the surrounding tasks.

MCP, or Model Context Protocol, is a standard way for an AI client to discover and use capabilities supplied by external servers. It reduces custom glue: a server describes what it offers, and a compatible client can call those capabilities through a consistent interface. MCP commonly exposes three concepts. **Tools** are actions, such as searching a repository, creating an issue, or controlling a browser. **Resources** are readable context, such as a file, record, or reference document. **Prompts** are reusable instruction templates supplied by a server. A client may support some or all of these, and permissions still matter; MCP does not make every action safe or automatically authorized.

Three concrete tool or connector tasks supported this Week 4 work. First, local workspace tools inspected the existing Week 3 files so the five FL-04 runs used real artifacts rather than invented examples. Second, browser control read the authenticated FlyRank assignment requirements and will submit the public deliverable links. Third, the GitHub web interface publishes the files to the existing public repository, providing version history and inspectable evidence. These are different capabilities, but an agent can coordinate them: context comes from the workspace and portal, a publishing action happens on GitHub, and the final submission happens in FlyRank.

A practical MCP upgrade to FL-04 would add a read-only link-checking tool. After the language-model review returns `READY`, the workflow would extract only the cited public URLs, request each one, and record the HTTP status, final destination, and check time. A broken or private link would automatically change the result to `REVISE`. The upgrade should remain bounded: it would not crawl unrelated pages, change repository permissions, or publish anything. The human would still inspect the final page visually and approve submission.

The main safety boundary is simple: reading is not the same as acting. An agent may analyze local drafts without much risk, but publishing a file or submitting a form changes external state. Those steps should use the narrowest permissions, display the exact destination, and run only with the user’s authorization. Secrets should never be copied into an artifact or prompt. Tool results also need skepticism because a successful call may still produce incomplete or misleading content.

My preferred design is therefore a small agent around a strict workflow. The agent selects the relevant files and tools; the FL-04 checklist produces a consistent recommendation; a link checker verifies basic availability; and a human approves publication. This gains useful coordination without giving up traceability. It is more modest than a fully autonomous system, but it matches the actual task and is easier to maintain.

Success would be measured by fewer broken submission links, consistent review notes, and less review time without an increase in false `READY` decisions. A short saved log would make those results auditable.
